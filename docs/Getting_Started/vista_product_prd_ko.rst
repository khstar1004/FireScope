VISTA 제품 기획서
======================

문서 개요
---------

- 문서명: ``VISTA 제품 기획서``
- 문서 성격: ``최종 제품 설명형 PRD``
- 기준일: ``2026-04-21``
- 문서 목적: VISTA가 어떤 제품인지, 누구를 위해 존재하며, 어떤 경험과 가치를 제공하는지를 일관된 제품 언어로 설명한다.
- 제품명 의미: ``Visualized Intelligence for Situational Tactical Awareness``


1. 제품 한 줄 정의
------------------

VISTA는 하나의 시나리오를 중심으로 ``배치``, ``시뮬레이션``, ``기록/재생``, ``브리핑``, ``실험`` 을 연결하는 시나리오 운영 워크벤치다.


2. 제품 소개
------------

VISTA는 단순한 지도 도구가 아니다. 사용자가 하나의 상황을 설계하고, 즉시 실행해 보고, 그 결과를 다시 설명하고, 다른 조건으로 반복 실험할 수 있게 만드는 제품이다.

이 제품의 중심에는 언제나 ``시나리오`` 가 있다. 사용자는 지도 위에 세력과 자산을 배치하고, 참조점과 임무를 정의한 뒤, 시간의 흐름에 따라 어떤 일이 벌어지는지 확인할 수 있다. 실행이 끝난 뒤에는 기록을 남기고 다시 재생할 수 있으며, 3D 브리핑 화면이나 실험 워크스페이스로 이어서 활용할 수도 있다.

즉, VISTA는 ``하나의 시나리오를 만들고, 돌리고, 읽고, 설명하고, 다시 검증하는 제품`` 이다.


3. 해결하려는 문제
------------------

복합적인 상황을 다루는 조직은 보통 네 가지 문제를 동시에 안고 있다.

- 상황을 빠르게 구성하기 어렵다.
- 구성한 내용을 즉시 실행해 보기 어렵다.
- 실행 결과를 설명 가능한 형태로 남기기 어렵다.
- 한 번 만든 시나리오를 브리핑, 교육, 분석, 실험에 재활용하기 어렵다.

VISTA는 이 문제를 ``시나리오 단일 원본`` 이라는 방식으로 해결한다. 하나의 시나리오를 제품 전체의 기준으로 삼아, 작성과 실행, 분석과 브리핑, 실험과 재검토가 한 흐름 안에서 이어지도록 설계한다.


4. 제품 비전
------------

VISTA의 비전은 다음 한 문장으로 정리할 수 있다.

``하나의 시나리오를 모든 운영 경험의 공통 원본으로 삼는 시나리오 기반 운영 플랫폼``

이 비전이 의미하는 바는 분명하다.

- 설계는 분리되지 않아야 한다.
- 실행은 따로 놀지 않아야 한다.
- 결과는 설명 가능해야 한다.
- 같은 시나리오는 여러 목적에 반복 활용될 수 있어야 한다.

VISTA는 기능을 늘리는 제품이 아니라, 시나리오를 중심으로 경험을 연결하는 제품을 지향한다.


5. 제품 포지셔닝
----------------

VISTA는 다음 세 가지 성격이 결합된 제품이다.

첫째, ``시나리오 편집기`` 다. 사용자는 지도 위에서 세력, 자산, 참조점, 임무를 배치하고 수정할 수 있다.

둘째, ``시간 기반 시뮬레이터`` 다. 작성한 시나리오를 바로 실행하고, 시간 흐름 속에서 상태 변화와 결과를 확인할 수 있다.

셋째, ``브리핑·분석·실험 플랫폼`` 이다. 동일한 시나리오를 기록·재생하고, 브리핑형 3D 화면으로 전환하며, 다른 조건으로 비교 실험할 수 있다.

따라서 VISTA의 가장 정확한 포지션은 ``시나리오 기반 운영 워크벤치`` 다.

이 제품은 단순 시각화 툴도 아니고, 대규모 엔터프라이즈 통합 지휘 시스템도 아니다. 그 사이에서 ``빠르게 만들고, 실행하고, 설명하고, 반복 검증하는 데 최적화된 제품`` 이다.


6. 핵심 사용자
--------------

시나리오 설계자
~~~~~~~~~~~~~~~

상황을 구성하고 자산을 배치하며 임무를 설계하는 사용자다. 이들에게 중요한 가치는 빠른 작성, 수정 용이성, 즉시 실행 가능성이다.

분석가
~~~~~~

실행 결과와 로그, 우세 변화, 잔존 전력, 결과 요약을 통해 시나리오의 의미를 읽는 사용자다. 이들에게 중요한 가치는 재현 가능한 기록과 읽기 쉬운 결과 표현이다.

교육·브리핑 운영자
~~~~~~~~~~~~~~~~~~

동일한 시나리오를 지도와 3D 화면으로 전환하며 설명하는 사용자다. 이들에게 중요한 것은 전달력과 몰입감, 그리고 설명 흐름의 자연스러움이다.

실험 사용자
~~~~~~~~~~~

같은 시나리오를 다른 조건으로 반복 실행하며 비교 검증하는 사용자다. 이들에게는 조건 조정의 용이성, 결과 비교 가능성, 재사용성이 중요하다.

데모·쇼케이스 운영자
~~~~~~~~~~~~~~~~~~~~

복잡한 설정을 깊게 보여주기보다, 하나의 시나리오를 통해 제품 가치와 장면 전환의 설득력을 보여주고 싶은 사용자다.


7. 핵심 가치 제안
-----------------

VISTA의 핵심 가치 제안은 다섯 가지다.

- ``한 번 만든 시나리오를 여러 방식으로 재사용할 수 있다.``
- ``편집과 실행이 한 흐름 안에 있다.``
- ``결과를 기록하고 다시 재생하며 설명할 수 있다.``
- ``2D 설계와 3D 브리핑이 같은 시나리오를 공유한다.``
- ``비교 실험과 반복 검증이 자연스럽게 이어진다.``

이 제품의 차별점은 개별 기능의 깊이보다, ``하나의 시나리오를 중심으로 여러 경험이 연결된다`` 는 구조적 일관성에 있다.


8. 핵심 경험
------------

시나리오 작성 경험
~~~~~~~~~~~~~~~~~~

사용자는 빈 시나리오나 준비된 예시에서 시작한다. 세력을 정하고, 자산을 배치하고, 참조점과 임무를 설정하면서 상황을 빠르게 구성할 수 있다.

시뮬레이션 경험
~~~~~~~~~~~~~~~~

작성한 시나리오는 곧바로 실행할 수 있다. 사용자는 시간 흐름을 따라 상태 변화와 주요 이벤트를 확인하고, 필요하면 일시정지하거나 단계별로 진행하며 흐름을 세밀하게 읽을 수 있다.

기록 및 재생 경험
~~~~~~~~~~~~~~~~~~

실행 결과는 기록으로 남길 수 있고, 이후 다시 불러와 재생할 수 있다. 이 기능은 단순 저장이 아니라 ``사후 분석과 설명`` 을 위한 핵심 경험이다.

브리핑 경험
~~~~~~~~~~~

같은 시나리오는 2D 지도에서 끝나지 않는다. 자산 소개형 3D 화면, 몰입형 브리핑 화면, 전장 관전형 3D 화면으로 전환되며, 사용자는 상황을 보다 전달력 있게 설명할 수 있다.

실험 경험
~~~~~~~~~

현재 시나리오는 다른 조건과 함께 반복 검증의 출발점이 된다. 사용자는 특정 과업이나 정책을 비교하거나, 자원 배치와 조합을 바꿔 보면서 더 나은 운영안을 탐색할 수 있다.

~~~~~~~~~~~~~~~~~

- 새 시나리오 생성
- 이름 변경
- 예시 시나리오 불러오기
- 외부 파일 업로드
- 시나리오 저장
- 저장된 시나리오 재호출

9.2 세력과 관계 관리
~~~~~~~~~~~~~~~~~~~~

- 세력 생성 및 수정
- 세력 간 관계 설정
- 세력별 구분과 시각적 식별
- 운용 기준 설정

9.3 자산 배치와 편집
~~~~~~~~~~~~~~~~~~~~

- 항공, 해상, 지상, 기지형 자산 배치
- 참조점 생성과 수정
- 자산 이동, 복제, 수정
- 임무 연결 및 상태 확인

- 임무와 자산의 연결 관리

9.5 시뮬레이션 실행 제어

- 연속 실행

- 속도 변경
- 실행 흐름 제어

9.6 기록과 리플레이

- 실행 기록 저장
- 기록 불러오기

- 조건 변경에 따른 반복 실행
- 비교 결과 확인
- 선정 결과 재활용


10. 대표 사용자 흐름
--------------------

10.1 시나리오 작성 후 실행
~~~~~~~~~~~~~~~~~~~~~~~~~~

사용자는 빈 시나리오에서 시작하거나 예시 시나리오를 불러온다. 세력과 자산을 배치하고 임무를 연결한 뒤, 시뮬레이션을 실행한다. 이후 로그와 결과 요약을 통해 시나리오의 의미를 확인한다.

10.2 실행 결과의 재생과 사후 분석
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

사용자는 실행 결과를 기록으로 남긴다. 나중에 다시 열어 단계별로 재생하면서, 어떤 시점에 어떤 변화가 일어났는지 다시 읽는다. 이 흐름은 분석과 보고, 교육 모두에 활용된다.

10.3 브리핑과 설명
~~~~~~~~~~~~~~~~~~

사용자는 지도에서 특정 자산이나 장면을 선택한 뒤 3D 브리핑 화면으로 전환한다. 동일한 시나리오를 보다 몰입감 있고 설명 친화적인 방식으로 전달할 수 있다.

10.4 반복 실험
~~~~~~~~~~~~~~

사용자는 현재 시나리오를 실험 워크스페이스로 연결하고, 조건을 바꿔 가며 여러 번 실행한다. 이후 비교 결과를 보고 더 적합한 운영안을 선택한다.


11. 정보 구조
-------------

VISTA의 정보 구조는 ``메인 편집기`` 와 ``체험형 서브앱`` 이 연결된 형태다.

- 메인 지도: 시나리오 작성과 실행의 중심 화면
- 기록/재생 영역: 실행 결과를 다시 읽는 분석 공간
- 3D 경험 계층: 자산 소개, 브리핑, 관전
- 실험 워크스페이스: 비교 실험과 반복 검증 공간

이 구조는 VISTA가 단일 작업만 수행하는 도구가 아니라, ``설계 -> 실행 -> 분석 -> 설명 -> 실험`` 을 하나로 연결하는 제품임을 보여준다.


12. 차별점
----------

VISTA의 차별점은 다음과 같다.

- 시나리오를 중심으로 여러 경험이 끊기지 않고 이어진다.
- 편집과 실행이 한 제품 안에서 바로 연결된다.
- 결과를 남기고 다시 읽는 경험이 강하다.
- 2D와 3D가 경쟁하는 것이 아니라 역할 분담을 이룬다.
- 실험 기능이 별도 연구 코드가 아니라 제품 경험 안에 포함된다.

많은 도구가 ``보기`` 또는 ``실행`` 또는 ``분석`` 중 한 축에 머무는 반면, VISTA는 이 축들을 하나의 시나리오로 연결한다는 점에서 차별화된다.


13. 도입 가치
-------------

VISTA를 도입하면 얻을 수 있는 가치는 명확하다.

- 시나리오 작성 속도가 빨라진다.
- 작성한 내용을 바로 검증할 수 있다.
- 결과를 설명 가능한 형식으로 남길 수 있다.
- 브리핑과 교육 자료로 재활용할 수 있다.
- 같은 시나리오를 기준으로 반복 실험이 가능해진다.

즉, VISTA는 단일 실행을 위한 도구가 아니라 ``운영 지식이 누적되는 도구`` 다.


14. 제품 경계
-------------

VISTA를 더 정확히 이해하려면 무엇이 아닌지도 분명해야 한다.

- 완전한 엔터프라이즈 통합 지휘 통제 시스템은 아니다.
- 모든 현실 요소를 정밀하게 복제하는 고정밀 시뮬레이터는 아니다.
- 실시간 협업과 권한 관리 중심의 대규모 운영 제품도 아니다.

VISTA의 강점은 ``정교한 현실 복제`` 보다는 ``빠른 구성, 실행, 설명, 반복 실험`` 에 있다.


15. 성공 지표
-------------

VISTA의 성공은 단순 사용량보다 다음 지표로 보는 것이 더 적절하다.

- 첫 시나리오 작성 후 첫 실행까지 걸리는 시간
- 저장된 시나리오의 재사용 비율
- 기록 생성 후 리플레이 진입 비율
- 브리핑형 3D 경험 진입 비율
- 실험 시작 후 정상 완료 비율
- 실험 결과를 다시 메인 경험으로 가져오는 비율
- 사용자 정의 시나리오 생성 비율

이 지표들은 VISTA의 핵심 가치인 ``재사용성``, ``연결성``, ``완결성`` 을 직접 반영한다.


16. 향후 발전 방향
------------------

앞으로 VISTA는 다음 방향으로 발전할 수 있다.

- 시나리오 공유와 협업 기능 강화
- 결과 비교와 보고 기능 강화
- 브리핑 템플릿과 출력 기능 강화
- 실험 환경의 폭 확장
- 2D와 3D의 상태 연결 강화
- 시나리오 기반 운영 지식의 축적 구조 강화

중요한 것은 기능의 양이 아니라, ``같은 시나리오를 더 많은 목적에 자연스럽게 재사용할 수 있게 만드는 것`` 이다.


17. 최종 정리
-------------

VISTA는 단순한 지도 편집 도구가 아니다. 동시에, 거대한 엔터프라이즈 운영 시스템도 아니다.

가장 정확한 정의는 다음과 같다.

``VISTA는 복합 시나리오를 빠르게 만들고, 실행하고, 기록하고, 설명하고, 다시 실험할 수 있게 해주는 시나리오 기반 운영 워크벤치다.``

이 제품의 가장 큰 강점은 기능 하나의 정교함보다도, ``하나의 시나리오를 중심으로 여러 경험을 연결한다`` 는 구조적 일관성에 있다. VISTA는 바로 그 일관성을 통해, 작성과 실행, 분석과 브리핑, 실험과 재검토를 하나의 제품 경험으로 묶어낸다.
