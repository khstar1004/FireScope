강화학습 설계 가이드
====================

문서 목적
---------

이 문서는 강화학습 설계를 "전장 시뮬레이션에서 승리확률을 높이기 위한 강화학습 설계 도구"로
이해하도록 돕기 위해 작성되었습니다.

시나리오 프리셋을 실제로 어떻게 RL에 연결해 쓸지에 대한 실전형 절차는
:doc:`../Gameplay/scenario_rl_playbook_ko` 문서를 함께 보면 됩니다.

핵심은 두 가지입니다.

- 최종 목표는 전장 승패에서 이길 확률을 높이는 정책을 찾는 것
- 현재 강화학습 설계는 그 목표를 바로 끝까지 푸는 완성품이 아니라, 그 목표를 향한
  첫 번째 학습 과제인 ``FixedTargetStrike`` 를 다루는 설계 도구라는 점


최종 목표와 현재 범위
--------------------

최종적으로 우리가 배우고 싶은 것은 단순한 평균 reward 최대화가 아닙니다.
진짜 목표는 다음에 가깝습니다.

- 주어진 전장 시나리오에서 임무를 완수할 확률을 높이는 것
- 아군 손실과 시간 지연을 감수하더라도 결국 승리하는 판단 구조를 배우는 것
- 특정 reward 조합이 아니라 실제 임무 성공률로 정책을 비교하는 것

하지만 현재 강화학습 설계가 직접 다루는 환경은 아직 더 좁습니다.

- 환경: ``blade/FixedTargetStrike-v0``
- 과제: 아군 항공기가 적 고정 표적을 협조 타격해 임무를 끝내는 문제
- 위치: ``gym/scripts/fixed_target_strike/train.py``

즉, 현재 강화학습 설계는 "전장 전체 승패"를 직접 최적화하는 완전한 전략 AI가 아니라,
"고정 표적 타격 임무 성공률"을 먼저 안정적으로 높이는 하위 과제를 푸는 구조입니다.

다만 이번 개선으로 현재 환경 안에서도 이전보다 더 높은 수준의 자원 배분을
학습할 수 있게 되었습니다.

- 예전: 모든 아군이 한 번에 하나의 공통 표적만 보도록 유도됨
- 현재: 각 아군이 자기 표적을 따로 고를 수 있어 ``자산-표적 배분`` 자체를 정책이 학습

즉, 아직 "초기 기지 배치"나 "편대 편성"까지 배우는 것은 아니지만,
최소한 현재 출격한 자산을 여러 표적에 어떻게 나눠 쓸지는 이제 학습 대상에 포함됩니다.


현재 구조가 왜 완전히 충분하지 않았는가
------------------------------------

이전 구조는 학습은 가능했지만, 최종 목적과는 몇 가지 어긋남이 있었습니다.

1. 평가와 저장 기준이 ``평균 reward`` 중심이었습니다.
   임무 성공률이 더 중요해도, reward shaping에 따라 다른 정책이 더 좋아 보일 수 있었습니다.

2. 알고리즘이 사실상 PPO 하나로 고정되어 있었습니다.
   시나리오에 따라 PPO가 늘 최선이라고 볼 근거는 부족했습니다.

3. 사용자가 결과를 "이 정책이 얼마나 자주 이기는가"로 읽기 어려웠습니다.
   reward 추이만 보면 승률 중심 비교가 어렵습니다.

4. 문서가 현재 범위와 최종 목표를 분리해서 설명하지 못했습니다.
   그래서 강화학습 설계 결과를 전체 전장 승리 정책으로 과대해석할 위험이 있었습니다.


이번 개선 사항
-------------

이번 개선으로 강화학습 설계는 아래 기준으로 동작합니다.

``success_rate`` 우선 평가
^^^^^^^^^^^^^^^^^^^^^^^^^^

- 체크포인트 평가와 최종 평가에서 ``success_rate`` 를 계산합니다.
- 최종 모델 선택 기준은 다음 순서입니다.

  - success rate가 높은 정책 우선
  - 같으면 mean reward가 높은 정책 우선
  - 그것도 같으면 평균 episode 길이가 더 짧은 정책 우선

즉, reward는 shaping 신호로 남기되, 최종 선택은 임무 성공률 중심으로 바꿨습니다.

다중 알고리즘 비교
^^^^^^^^^^^^^^^^^^

이제 강화학습 설계는 같은 시나리오에 대해 여러 알고리즘을 순차 비교할 수 있습니다.

- ``PPO``
- ``A2C``
- ``SAC``

사용자는 한 개만 선택해서 단일 실험을 돌릴 수도 있고, 여러 개를 함께 선택해
비교 실험을 돌릴 수도 있습니다.

알고리즘별 진행 상황과 최종 비교 결과는 요약 JSON과 강화학습 설계 UI에서 함께 확인할 수 있습니다.

승리확률 중심 시각화
^^^^^^^^^^^^^^^^^^^^

UI에서 다음을 바로 볼 수 있습니다.

- 평가 승리확률 추이
- 평가 reward 추이
- episode reward 추이
- 알고리즘별 최종 win rate
- 어떤 알고리즘이 최종 선택되었는지


강화학습 설계를 어떻게 이해해야 하는가
------------------------------------

현재 강화학습 설계를 사용할 때는 아래처럼 생각하는 것이 가장 정확합니다.

- 강화학습 설계는 전장 전체를 끝까지 푸는 완성형 지휘 AI가 아니다
- 대신 "임무 성공률을 명시적으로 높이는 학습 파이프라인"을 갖춘 설계 도구다
- 여기서 잘 되는 보상 설계, 시나리오 구성, 알고리즘 후보를 찾은 뒤
  더 넓은 전장 과제로 확장해야 한다

따라서 강화학습 설계 결과는 다음 용도로 특히 적합합니다.

- 고정 표적 타격 교리 비교
- 생존성 대 공격성 trade-off 실험
- TOT 보상이 실제 성능 개선으로 이어지는지 검증
- 어떤 알고리즘이 현재 임무에 더 잘 맞는지 비교


빠른 시작
---------

필수 환경
^^^^^^^^^

- Node 20.19.x
- npm 10.8.x
- Python 3.12

설치
^^^^

프런트엔드 준비::

   cd client
   npm install

Gym 패키지와 RL 의존성 준비::

   cd gym
   pip install -e .[gym]

실행
^^^^

보통은 아래 명령으로 충분합니다::

   cd client
   npm run standalone

또는 개발 모드::

   cd client
   npm run start


UI 사용 절차
------------

1. 지도 화면에서 ``강화학습 설계`` 로 이동합니다.
2. 기본 시나리오, 현재 지도 시나리오, 업로드 JSON 중 하나를 불러옵니다.
3. ``초심자 빠른 시작`` 영역에서 자동 분석 결과를 확인합니다.
4. ``시나리오 카탈로그`` 에서 아군 항공기, 표적, 고가치 표적을 선택합니다.
5. ``학습 설정`` 에서 알고리즘 하나 또는 여러 개를 선택합니다.
6. timesteps, eval episodes, progress eval 값을 조정합니다.
7. ``보상 계수 설계`` 에서 원하는 전술 경향을 조절합니다.
8. ``학습 시작`` 을 누릅니다.
9. 결과 그래프, 최신 평가 요약, 알고리즘 비교 결과를 확인합니다.
10. 선택된 최종 모델 ZIP, 요약 JSON, 평가 시나리오, 평가 리플레이를 다운로드하거나
    지도에서 다시 엽니다.


알고리즘은 어떻게 고르면 좋은가
------------------------------

``PPO``
^^^^^^^

- 현재 기본 추천값입니다.
- 대부분의 입문 실험에서 기준선으로 삼기 좋습니다.

``A2C``
^^^^^^^

- PPO보다 단순한 기준선을 보고 싶을 때 좋습니다.
- 빠르게 비교선을 하나 더 만들고 싶을 때 유용합니다.

``SAC``
^^^^^^^

- 연속 action 탐색이 더 필요한 경우 비교 후보로 볼 수 있습니다.
- 다만 짧은 timesteps에서는 PPO보다 불안정하게 보일 수 있습니다.

권장 실험 순서는 다음과 같습니다.

1. ``PPO`` 단독으로 기준선 확보
2. 같은 시나리오에서 ``PPO + A2C`` 비교
3. 필요할 때 ``SAC`` 를 추가
4. 승률과 리플레이를 함께 보고 최종 정책 선택


승리확률 비교를 제대로 하려면
---------------------------

``evalEpisodes`` 가 1이면 결과가 너무 요동칠 수 있습니다.
승률 중심 비교를 하려면 다음을 권장합니다.

- 빠른 스모크: ``evalEpisodes = 1``
- 비교용 기준선: ``evalEpisodes = 3 ~ 5``
- 더 믿을 만한 비교: ``evalEpisodes = 5`` 이상

체크포인트 평가도 너무 드물면 추세를 읽기 어렵고,
너무 잦으면 학습 시간이 늘어납니다.
작은 시나리오에서는 ``progressEvalFrequency = 256 ~ 512`` 정도가 무난합니다.


보상 계수는 어떻게 해석해야 하는가
---------------------------------

보상은 최종 목적 그 자체가 아니라, 승리확률을 높이도록 유도하는 shaping 신호입니다.

``Kill Base``
^^^^^^^^^^^^^

- 표적 제거 자체를 얼마나 강하게 밀지 결정합니다.

``High Value Bonus``
^^^^^^^^^^^^^^^^^^^^

- 공군기지, 지휘소 같은 전략 표적을 더 우선하도록 만듭니다.

``TOT Weight`` / ``TOT Tau Seconds``
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

- 여러 타격의 동시성을 얼마나 중요하게 볼지 결정합니다.

``Threat Step Penalty``
^^^^^^^^^^^^^^^^^^^^^^^

- 위협 지역 체류를 얼마나 싫어하게 만들지 결정합니다.

``Launch Cost``
^^^^^^^^^^^^^^^

- 무장 낭비를 억제합니다.

``Time Cost``
^^^^^^^^^^^^^

- 오래 끄는 정책을 억제합니다.

``Loss Penalty``
^^^^^^^^^^^^^^^^

- 아군 손실을 얼마나 크게 벌점화할지 정합니다.

``Success Bonus`` / ``Failure Penalty``
^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^

- 임무 성공과 실패를 terminal reward에 직접 반영합니다.
- 최종 목적이 승리확률이라면 이 두 값은 reward shaping 안에서도 특히 중요합니다.


결과 읽는 법
------------

``평가 승리확률 추이``
^^^^^^^^^^^^^^^^^^^^^^

- 체크포인트마다 deterministic evaluation success rate를 보여줍니다.
- 지금 구조에서 가장 먼저 봐야 하는 그래프입니다.

``평가 보상 추이``
^^^^^^^^^^^^^^^^^^

- reward shaping이 어떤 방향으로 움직이는지 보여줍니다.
- 승률은 같은데 reward만 크게 다르면 보상 설계를 다시 볼 근거가 됩니다.

``알고리즘 비교``
^^^^^^^^^^^^^^^^^

- 각 알고리즘의 win rate, mean reward, 종료 사유를 비교합니다.
- 최종 선택 모델은 여기서 가장 승률이 높았던 알고리즘입니다.

``Reward breakdown``
^^^^^^^^^^^^^^^^^^^^

- 최신 평가에서 kill, TOT, threat, loss, terminal bonus가 어떻게 구성됐는지 보여줍니다.
- reward는 좋아졌는데 win rate가 안 오르면 shaping이 잘못됐을 가능성을 의심해야 합니다.


산출물 위치
-----------

학습 결과는 작업별 디렉터리에 저장됩니다.::

   .firescope_rl_jobs/<job-id>/

대표 파일은 다음과 같습니다.

- ``scenario.json``: 실제 학습에 사용한 시나리오
- ``summary.json``: 알고리즘 비교 결과와 최종 선택 정보
- ``progress.json``: 중간 체크포인트와 진행 상태
- ``fixed_target_strike_policy.zip``: 최종 선택 정책
- ``runs/<algorithm>/...``: 알고리즘별 개별 모델과 평가 결과
- ``eval_scenario.json``: 최종 선택 정책의 평가 시나리오
- ``eval_recording.jsonl``: 지도에서 다시 볼 수 있는 평가 리플레이


직접 스크립트로 실행하는 방법
----------------------------

UI 대신 스크립트로 직접 돌릴 수도 있습니다.::

   cd gym
   .\.venv\Scripts\python.exe scripts\fixed_target_strike\train.py ^
     --algorithms ppo a2c sac ^
     --timesteps 4096 ^
     --max-episode-steps 240 ^
     --eval-episodes 5 ^
     --progress-eval-frequency 512 ^
     --progress-eval-episodes 2 ^
     --controllable-side-name BLUE ^
     --target-side-name RED ^
     --ally-ids blue-striker-1 blue-striker-2 ^
     --target-ids red-sam-site red-airbase ^
     --high-value-target-ids red-airbase


현재 한계와 다음 확장 방향
------------------------

현재 구조가 개선되었더라도 아직 다음 한계는 남아 있습니다.

- 과제가 ``FixedTargetStrike`` 에 한정되어 있습니다.
- 적 행동 다양성, 장기 캠페인 상태, 다단계 임무 전환은 충분히 반영하지 못합니다.
- "전장 전체 승리"를 직접 학습하려면 더 넓은 state, action, terminal condition이 필요합니다.

다음 단계로는 이런 확장이 필요합니다.

- 더 넓은 전장 승패 조건을 가진 환경 추가
- 적 force composition과 대응 전략 다양화
- 임무 단계 전환이 포함된 계층형 정책
- 시나리오 묶음 평가로 일반화 성능 확인
